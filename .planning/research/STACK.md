## 1. Action-Level Residual MAPPO

**Verdict: No new library. Pure code change inside `mappo.py` and `local_mappo_backend.py`.**
**Confidence: HIGH**

### What it is

Action-level residual means: at each step, Max-Pressure computes a baseline action `a_mp`. MAPPO's Actor observes `(obs, a_mp)` — the MP action is concatenated to the observation vector — and outputs a correction distribution over actions. The policy is incentivized to output `a_mp` by default and deviate only when it expects improvement.

This differs from the existing `residual_mp` reward mode (which only penalizes deviation in the reward signal). Action-level residual augments the **input** to the Actor, so the network can condition its decisions on what MP would have done.

### Implementation pattern

```python
# In local_mappo_backend.py, before each act() call:
mp_actions = max_pressure.actions(obs, env=env)

# Encode MP action as one-hot and append to obs:
def augment_obs_with_mp(obs_arr, mp_action, n_actions):
    one_hot = np.zeros(n_actions, dtype=np.float32)
    one_hot[mp_action] = 1.0
    return np.concatenate([obs_arr, one_hot])

# Pass augmented obs to trainer.act_batch(...)
# Actor input dim increases by n_actions (max 4 for this network)
```

The `Actor` and `Critic` in `mappo.py` take `obs_dim` as a constructor argument — changing `obs_dim` to `obs_dim + n_actions` is a single-parameter change. The existing `MAPPOTrainer.__init__` signature already supports this without modification.

### What NOT to do

- Do not use imitation learning (behavioral cloning pre-training) as a substitute. The project explicitly chose residual over IL because IL requires a separate BC phase and collapses the policy toward MP rather than learning selective correction.
- Do not use any external residual RL library (e.g., `garage`, `rlpyt`, `stable-baselines3`). They would require rewriting the custom MAPPO stack and add heavy dependencies not compatible with the existing Pixi environment.
- Do not add `a_mp` to the observation in the Critic. The Critic uses global obs (concatenated per-agent observations). Adding MP actions to the Critic input introduces MP-dependent value bootstrapping that can destabilize training.

### Why the existing codebase supports this cleanly

`MaxPressureController` already runs on every training step at line 132 of `local_mappo_backend.py` (to compute `mp_deviation_by_agent` for `residual_mp` rewards). The MP call is already paid for — augmenting the observation costs no additional SUMO queries.

## 2. Curriculum Training

**Verdict: No new library. Curriculum = a sorted list of `.rou.xml` paths passed to the existing training loop.**
**Confidence: HIGH**

### What it is

Curriculum training here means: train across progressively more varied demand scenarios rather than on a single static `demand.rou.xml`. The demand files are generated by the headless demand studio CLI (see Section 4). The training loop selects a demand file per episode according to a curriculum schedule.

### Implementation pattern

The existing `TrainConfig.route_file` is a single string. The change required is:

1. Add `route_files: list[str]` to `TrainConfig` (alongside the existing `route_file`).
2. In the training loop (`local_mappo_backend.py`), select `route_files[ep % len(route_files)]` per episode (round-robin), or use a deterministic schedule (easy-to-hard ordering by demand scale).
3. Re-create the SUMO environment per episode with the selected route file.

The environment is already re-created per episode (`env.reset()` reseeds; the env object is created once before the episode loop). The change requires creating a new env object per episode when the route file changes — a minor refactor.

### Curriculum schedule recommendation

Use **round-robin** over a list of 5-10 demand files covering: AM peak, PM peak, off-peak, weekend, and 2-3 scaled variants (0.7x, 1.0x, 1.3x of nominal). This is the minimum viable curriculum that exposes MAPPO to irregular demand without complex scheduling logic.

Do not implement self-paced or adaptive curriculum (selecting scenarios based on current policy performance). That requires a separate difficulty estimator and adds 200+ lines of scaffolding for marginal benefit given the 30-episode training budget and RTX 4070 time constraint.

### What NOT to do

- Do not use `stable-baselines3` or `RLlib` curriculum callbacks — they require adopting their training loops, which would mean abandoning the custom MAPPO implementation.
- Do not use `Curriculum Learning` libraries like `syllabus` or `TeachMyAgent` — they are designed for Gymnasium-compatible environments with centralized training loops. SUMO's TraCI lifecycle (one process per environment) makes multi-environment curriculum impractical on a single laptop GPU.

## 3. Parquet Dataset Logging

**Verdict: Use `pyarrow 20.0.0` (already added to `pixi.toml`). Use `pyarrow.parquet.ParquetWriter` for streaming writes.**
**Confidence: HIGH** — verified by installing pyarrow 20.0.0 and running the write pattern in the actual Pixi environment.

### Library

| Library | Version | Source | Purpose |
|---------|---------|--------|---------|
| pyarrow | 20.0.0 | conda-forge | Parquet read/write backend |

pyarrow is the only option. `fastparquet` is the alternative but:
- It is slower for columnar writes
- PIRA uses `pd.read_parquet()` which auto-selects pyarrow
- pyarrow is the standard backend for Apache Arrow, which is what pandas 3.x uses internally for `to_parquet()`

### Schema (matches PIRA expectations exactly)

From `PIRA.ipynb` (origin/PIRA branch), PIRA reads:

```python
df.groupby(['scenario_id', 'time_step'])
group[['queue_ns', 'queue_ew', 'avg_speed_ns', 'avg_speed_ew']]  # features
group[['delay', 'throughput', 'queue_total']]                    # targets
```

This maps exactly to `ASCE_DATASET_COLUMNS` in `ece324_tango/asce/schema.py`. No schema change is needed — only the output format changes from CSV to Parquet.

### Recommended write pattern

Use `pyarrow.parquet.ParquetWriter` for incremental streaming writes during evaluation (not batched end-of-episode writes). This is important because SUMO can terminate early with `FatalTraCIError` — a streaming writer survives partial episodes whereas `df.to_parquet()` at the end would lose the episode's data.

```python
import pyarrow as pa
import pyarrow.parquet as pq

# Define schema once (matches ASCE_DATASET_COLUMNS)
PARQUET_SCHEMA = pa.schema([
    ("intersection_id", pa.string()),
    ("time_step", pa.float32()),
    ("queue_ns", pa.int32()),
    ("queue_ew", pa.int32()),
    ("arrivals_ns", pa.int32()),
    ("arrivals_ew", pa.int32()),
    ("avg_speed_ns", pa.float32()),
    ("avg_speed_ew", pa.float32()),
    ("current_phase", pa.int32()),
    ("time_of_day", pa.float32()),
    ("action_phase", pa.int32()),
    ("action_green_dur", pa.float32()),
    ("delay", pa.float32()),
    ("queue_total", pa.int32()),
    ("throughput", pa.int32()),
    ("scenario_id", pa.string()),
    ("controller", pa.string()),   # add: "mappo" | "fixed_time" | "max_pressure"
])

# Open writer once per eval run
writer = pq.ParquetWriter("data/final/dataset.parquet", PARQUET_SCHEMA)

# Write a batch per step (or buffer N rows and flush)
batch = pa.RecordBatch.from_pydict({col: [val] for col, val in row.items()},
                                    schema=PARQUET_SCHEMA)
writer.write_batch(batch)

# Close after all episodes
writer.close()
```

The `controller` column should be added to the schema beyond what `ASCE_DATASET_COLUMNS` currently specifies. PIRA needs to distinguish training data generated by different controllers when building graph snapshots. This column is not in `schema.py` yet — it needs to be added.

### What NOT to do

- Do not use `pandas.DataFrame.to_parquet()` as the primary logging path. It requires accumulating all rows in RAM before writing and will OOM on long curriculum runs (30 episodes x 8 agents x 300s / 5s = 14,400 rows per run — manageable, but streaming is more robust).
- Do not use `pyarrow.dataset.write_dataset()` with `partition_cols=['scenario_id']`. PIRA reads a single flat parquet file at `data/final/dataset.parquet`. Partitioned datasets require a directory path and more complex read logic in PIRA.
- Do not use `h5py` or `.npz` formats. PIRA explicitly calls `pd.read_parquet(parquet_path)` with Parquet — switching formats requires changing PIRA.

## 4. Headless Demand Generation

**Verdict: Add dash, plotly, lxml, pyproj as PyPI dependencies. Call `generate_scenario()` directly from a thin CLI wrapper.**
**Confidence: MEDIUM** — function is confirmed callable, but `NETWORK_CACHE` is module-level state that requires `load_network_summary()` to be called before `generate_scenario()`.

### Required new dependencies

| Library | Version | Purpose | Notes |
|---------|---------|---------|-------|
| dash | >=2.18 | Required import in `app.py` | Not needed for headless use, but `app.py` imports it at module level |
| plotly | >=5.24 | Required import in `app.py` | Same — module-level import |
| lxml | >=5.3 | XML generation for `.rou.xml` | Used in `generate_scenario()` for `etree.Element()` |
| pyproj | >=3.6 | Coordinate projection | Used when mapping TMC locations to SUMO junctions |

These are all available as PyPI packages. Add to `[pypi-dependencies]` in `pixi.toml` (consistent with how `sumo-rl` and `python-dotenv` are declared).

### Why Dash/Plotly are required even for headless use

`app.py` imports `from dash import Dash, ...` and `import plotly.graph_objects as go` at module level (lines 27-30). There is no headless import path that avoids these. The cleanest approach is to import only the functions needed from `app.py` rather than importing the whole module, which still triggers module-level initialization including `TMC_DF = load_tmc_data(TMC_PATH)` and `NETWORK_CACHE`. This module-level initialization is safe (it reads a CSV and populates a dict).

### Calling `generate_scenario()` headlessly

```python
# In new ece324_tango/modeling/generate_demand.py
import sys
sys.path.insert(0, str(REPO_ROOT / "apps" / "demand_studio"))
from app import generate_scenario, load_network_summary, NETWORK_CACHE

# Step 1: populate NETWORK_CACHE (required before generate_scenario)
summary = load_network_summary(Path("sumo/network/osm.net.xml"))
network_store = {"network_key": str(Path("sumo/network/osm.net.xml").resolve())}

# Step 2: call generate_scenario with desired params
result = generate_scenario(
    network_store=network_store,
    selected_locations=[],          # empty = use all TMC locations
    global_demand_scale=1.0,
    time_window_start=480,          # 8:00 AM in minutes
    time_window_duration=60,
    simulation_begin=0,
    simulation_end=3600,
    ...                             # other params from app.py signature
)

if result.success:
    route_file = result.demand_path  # Path to generated .rou.xml
```

`generate_scenario()` returns a `GenerationResult` dataclass with `demand_path: Path | None` pointing to the generated `.rou.xml` file. That path feeds directly into `TrainConfig.route_file`.

### What NOT to do

- Do not copy `generate_scenario()` out of `app.py` into a separate module. The function depends on `NETWORK_CACHE`, `TMC_DF`, and over a dozen helper functions defined in `app.py`. Extracting it would require replicating ~600 lines of supporting code.
- Do not spawn `app.py` as a subprocess and scrape the Dash server. The function is directly callable and has a clean return type.
- Do not reinstall `lxml` via conda (it exists as a PyPI package and `pixi add --pypi lxml` matches the existing pattern for PyPI deps in `pixi.toml`).

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Parquet backend | pyarrow 20.0.0 | fastparquet | pyarrow is faster, is the pandas 3.x default, and PIRA already assumes `read_parquet` works with it |
| Residual RL | Custom obs augmentation | stable-baselines3 residual wrapper | Would require abandoning custom MAPPO; no MAPPO wrapper exists in SB3 for multi-agent settings |
| Curriculum scheduling | Round-robin over file list | Self-paced curriculum (syllabus, TeachMyAgent) | Over-engineered for a 30-episode budget; adds 500+ LOC for marginal benefit |
| Demand headless | Import `generate_scenario` from `app.py` | Rewrite demand generation separately | app.py is 3031 lines of tested code; reimplementing is 2+ days of work |
| Demand headless | Import `generate_scenario` from `app.py` | HTTP calls to running Dash server | Fragile; requires starting/stopping a server process around training runs |
