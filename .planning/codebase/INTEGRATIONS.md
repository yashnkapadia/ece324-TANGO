# External Integrations

**Analysis Date:** 2026-03-29

## APIs & External Services

**Traffic Simulation:**
- SUMO (Simulation of Urban Mobility) 1.26.0
  - What it's used for: Multi-agent traffic simulation with TraCI (Traffic Control Interface)
  - SDK/Client: `sumo-rl` Python package v1.4.5+
  - Entry points:
    - `ece324_tango/asce/env.py` - `SumoEnvironment` initialization
    - `ece324_tango/asce/trainers/local_mappo_backend.py` - SUMO stepping and state extraction
  - TraCI exceptions handled: `traci.exceptions.FatalTraCIError` (line 8 in local_mappo_backend.py)
  - Communication: Direct subprocess via TraCI (libsumo or TraCI server)

**Network Data:**
- OpenStreetMap (OSM)
  - What it's used for: Toronto corridor road network and topology
  - Format: OSM XML imported, converted to SUMO network format
  - Files:
    - `sumo/network/osm_bbox.osm.xml.gz` - Raw OSM extract (281 KB)
    - `sumo/network/osm.net.xml` - Converted SUMO network (3.1 MB, 29K lines)
  - Authority: SUMO netconvert v1.26.0 (OSM conversion)
  - No live API - static network snapshot for reproducibility

**Traffic Demand:**
- Turning Movement Count (TMC) Calibration
  - What it's used for: Real-world traffic volume calibration
  - Source: Toronto TMC data (referenced in README as "TMC-calibrated demand")
  - File: `sumo/demand/demand.rou.xml` (10 KB, 70 car flows)
  - Status: Current pipeline uses nom (car-only) demand
  - Future: data-setup branch contains multi-modal (transit, pedestrian) demand generation

## Data Storage

**File-Based Persistence:**
- Local filesystem only (no database)
- Training artifacts:
  - `models/` - Trained PyTorch model checkpoints (.pt files)
    - Path constant: `ece324_tango.config.MODELS_DIR`
    - Example: `models/asce_mappo_toronto_demand.pt`
- Evaluation results:
  - `reports/results/` - Evaluation CSV metrics
  - Path constant: `ece324_tango.config.RESULTS_DIR`
  - Example: `asce_eval_metrics_toronto_demand.csv`
- Training logs:
  - `data/processed/` - Rollout samples and processed data
  - Path constant: `ece324_tango.config.PROCESSED_DATA_DIR`
  - Example: `asce_rollout_toronto_demand.csv`

**CSV Schema:**
- Rollout data: `ASCE_DATASET_COLUMNS` from `ece324_tango/asce/schema.py`
  - Columns: intersection_id, time_step, queue_ns, queue_ew, arrivals_ns, arrivals_ew, avg_speed_ns, avg_speed_ew, current_phase, time_of_day, action_phase, action_green_dur, delay, queue_total, throughput, scenario_id
  - Generated by: `LocalMappoBackend.train()` in `ece324_tango/asce/trainers/local_mappo_backend.py:200+`
- Episode metrics: mean_global_reward, seed, scenario_id (from train)
  - Generated by: Training episode loops
- Evaluation metrics: controller, person_time_loss_s, avg_trip_time_s, arrived_vehicles, vehicle_delay_jain
  - Generated by: Evaluation loop in `predict.py`

**Caching:**
- None currently implemented
- Observations normalized on-the-fly via Welford algorithm in `ece324_tango/asce/obs_norm.py`

## Baseline Controllers

**Local Implementation (No External Service):**
1. **Fixed-Time Controller** - Cycle phases uniformly
   - Class: `FixedTimeController` in `ece324_tango/asce/baselines.py`
   - Input: action_size_by_agent (number of phases per intersection)
   - Output: Deterministic phase sequence (0, 1, 2, ... mod num_phases)

2. **Max-Pressure Controller** - Queue-responsive phase selection
   - Class: `MaxPressureController` in `ece324_tango/asce/baselines.py`
   - Input: Current observations, SUMO environment reference
   - Algorithm: Pressure(phase) = sum of (inflow_queue - outflow_queue) for green movements
   - SUMO integration: Calls `env.sumo.trafficlight.getControlledLinks(ts_id)` for lane-level state
   - Output: Phase index with highest pressure

3. **MAPPO (Multi-Agent PPO)** - Learned policy
   - Framework: PyTorch
   - Architecture: Shared-parameter actor + centralized critic
   - Training: In `ece324_tango/asce/trainers/local_mappo_backend.py`
