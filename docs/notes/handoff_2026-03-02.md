## 4) Root-cause findings from warning/correctness investigation

### 4.1 Confirmed warning root cause in metrics extraction
- Existing code attempted `traci.edge.getShape`, which is not present in current TraCI edge domain.
- Verified by inspecting installed TraCI module and sumo-rl reference code:
  - `LaneDomain.getShape` exists.
  - `EdgeDomain.getShape` does not.
- Fix applied via lane-shape voting.

### 4.2 Confirmed real correctness issue on Toronto topology
- Shared local MAPPO actor assumed uniform obs vector length across intersections.
- Toronto network has heterogeneous controlled-lane counts -> obs size mismatch -> actor matmul shape crash.
- Fixed via dynamic max-dim discovery + zero padding before actor input.

### 4.3 Confirmed evaluate-path robustness issue
- `LocalMappoBackend.evaluate` created env before validating checkpoint existence.
- Fixed: early path validation + improved close behavior.

### 4.4 Long-horizon Toronto failure still open
- At longer horizon (`seconds=600`) in controllable TLS mode, runs can fail with:
  - `FatalTraCIError: Connection closed by SUMO`
- Observed around ~50-57 control steps.
- This reproduces even with fixed/constant actions (not clearly MAPPO-specific).
- Direct SUMO CLI run on same net/route reaches horizon successfully.
- `fixed_ts=True` path in sumo-rl avoids crash (but disables learning/control).
- Current best hypothesis: issue in SUMO-RL controllable TLS interaction on this corridor/net configuration.

## 5) Proposal alignment and objective interpretation
From proposal ([TANGO-proposal.tex](/home/as04/ece324-TANGO/TANGO-proposal.tex)):
- Phase 1 success criterion is delay reduction:
  - `Delay_MAPPO <= 0.90 * Delay_Baseline`
  - Baseline = better of Fixed-Time or Max-Pressure
  - Measured over 10 seeds
- Delay is defined as SUMO `timeLoss`, person-weighted using occupancy assumptions (vehicles/TTC).
- Throughput and Jain fairness are secondary evaluation metrics, not the primary success criterion.

Current code status:
- Supports `reward_mode=objective|sumo`.
- Logs proposal KPIs in eval output:
  - `time_loss_s`
  - `person_time_loss_s`
  - `avg_trip_time_s`
  - `arrived_vehicles`

Recommendation for defensibility:
- Train reward can be composite (delay/throughput/fairness) for stability shaping.
- Final project claim should be based on proposal criterion using `time_loss_s` (and preferably person-weighted version), compared against best baseline over 10 seeds.
